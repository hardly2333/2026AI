### 一.实验题目

一阶逻辑的归结推理

### 二.实验内容

编写函数 `ResolutionProp`实现命题逻辑的归结推理. 该函数要点如下:

* 输入为子句集(数据类型与格式详见课件), 每个子句中的元素是原子命题或其否定.
* 输出归结推理的过程, 每个归结步骤存为字符串, 将所有归结步骤按序存到一个列表中并返回, 即返回的数据类型为 `list[str]`.
* 一个归结步骤的格式为`步骤编号 R[用到的子句编号] = 子句`. 如果一个字句包含多个公式，则每个公式用编号`a,b,c...`区分，如果一个字句仅包含一个公式，则不用编号区分.

##### 1.算法原理

事实上，要完成归结推理，核心在于使用反证法寻找子句集中的不一致。
笔者将该实验拆分成三个大的模块：**输入数据处理、MGU算法实现、归结算法**
接下来笔者将一一分析：

###### 一. 输入数据处理
由于输入的是子句集， 我们沿着 **子句集->子句->原子公式->函数/变量/常量** 的路径思考，反向来逐步编写函数。
首先，我们要处理 **函数/变量/常量/**。
关于变量、常量的区分，我们采用直接采用匹配的方式。笔者在这里选择了 x, y, z, u, v, w, xx, yy, zz, uu, vv, ww
这些字母作为变量。值得注意的是，由于笔者会在后面将两个子句中的x区分，改写成x\_1,x\_2，所以笔者还把这些名字添加到匹配表中。
关于函数和其他两者的区分，我们用parse\_term函数，若为常量或者变量，则直接返回；若为函数，则将其参数列表提取出来，递归调用该函数。

接着，我们需要处理 **原子公式**
我们知道原子公式的形式是: P(t1,t2,...,tn) <br>
根据课件，我们需要将其分解为(符号，谓词，参数)的形式，需要编写parse\_atom\_formula函数。由于符合和谓词形式比较固定，我们先将这两者先分离出来，再来处理比较灵活的参数。<br>
由于可能存在 P(a,f(x,y))的类似形式，我们不能直接用逗号分割后用parse\_term处理。由此，我们引入split\_args函数，通过一个类似
括号匹配的机制将其分割出来，之后便可使用parse\_term处理后并合并成tuple类型。

再者，我们处理 **子句**
子句的形式一般是: (P(x),~Q(y),) ，不难观察发现，split\_args函数仍可用于子句中各个原子公式的分割，公式内部的函数恰好由于公式的左括号的原因不会被分割。 由此，我们简单处理掉括号和逗号就可以使用split\_args函数分割出原子公式，通过调用parse\_atom\_formula函数，再append到我们给定的tuple中，就可以得到处理后的clause。

最后，我们来处理 **子句集**
子句集需要set()形式，我们前面的函数保证其子句会返回tuple形式，所以我们只要把每一个kb中的clause调用parse\_clause函数，得到的结果添加进set()就可以了。

###### 二. MGU算法实现
给定两个原子公式，MGU算法可以大致分为以下步骤：
1.对两个原子公式应用替换字典
2.检查两个公式是不是相同，若相同，不需要替换了，直接该字典
3.如果有一个公式是单变量（可以替换），我们就调用unify\_var函数，尝试替换，并返回新字典
4.如果两个都是有多参数的，若要能够替换到相同，必须要有相同的谓词和参数个数。如果得以相同，对每个参数尝试替换
5.返回第一步，直到成功或失败

根据以上步骤，我们需要编写三个函数： **apply\_dict-应用替换字典，unify-进行主循环，unify\_var-尝试变量替换、增添字典**

首先，我们来处理**apply\_dict**。
我们输入的是一个原子公式，而最终需要替换掉 x:a f(x,x):a。 那么我们就需要递归调用处理。
由此，若term是一个变量且在字典中，则获取相应替换项；若为复合项（原子公式或函数），就递归对每一项尝试应用替换，并重新添加成tuple。

接着，我们处理**unify**。
该函数执行MGU算法的主循环逻辑。第一个步骤调用上面的apply\_dict，第二到四步则使用if\_else处理。
选择判断进行以下检测：
若两项完全相同，则当前的 substitution 已经满足要求。<br>
若其中一项为变量，则通过 unify\_var 尝试建立绑定关系。<br>
若两项均为复合结构，先区分 原子公式（3元组） 和 函数项（2元组）。若匹配成功，则通过 zip 函数同步遍历参数列表，并将前一对参数产生的置换结果作为下一对参数合一的输入。确保了同一个变量在整个公式中的约束是一致的。

再者，我们处理**unify\_var**。
该函数负责在字典中增添新的变量绑定。该函数的编写有点类似于并查集的编写。若已有绑定，我们递归寻找最终值。同时，为了字典的简洁性，我们每次添加新绑定是将新的约束应用到旧的绑定值中，提高了效率。<br>

此外，由于可能会出现 x= f(x)这种特殊情况，我们需要编写occur\_check进行回环检查，防止进入死循环。
该函数会通过遍历所有参数项，确保不存在自引用的情况。

##### 三.归结算法
由于实际给到计算机的子句集已经包括取反后的结论了，所以我们只需要反复调用单步归结，一直到得到空语句或者无法得到空语句就可以论证正确与否了。

那么我们需要实现 单步归结single-step函数，主循环solve函数，和一些用来完善输出的辅助函数。

首先是单步归结函数：
为避免不同子句的同名变量在MGU中被认为同一个，我们调用rename-clause函数为其变量加上后缀。加上后缀后，我们通过双重循环遍历两个子句的文字，当谓词相同符号相反时，尝试用mgu合一。若成功，就需要移除掉这两项，并将新的字典应用到剩下所有项中。

接着，我们要处理主循环solve:
我们需要先创建一个history列表来记录类似推理步骤的“日志”，将前提都添加进去。接着，我们将所有前提先打印出来，再进行反复归结。
我们一开始采用两重循环，让每个新的子句都跟旧的子句进行单步归结，避免遗漏。针对得到的结果，如果能够进行归结，我们先排除重复，
随后调用一些输出相关的辅助函数，实现该步骤记录与打印。一直到归结结果为空时结束循环过程。

再者，关于一些完善输出的辅助函数：
我们用clean—name和format-term函数来除去前面添加的后缀(x\_1中的\_1)；用get\_clause生成推导的标签，根据长度选择是回复1a还是1之类的； 用format\_clause负责最终子句呈现，单文字返回tuple形式，多文字返回(L1,L2)形式。

def unify_var(var, term, substitution):
# step3-4's func, adding new substitution
# unify var with term, return new substitution or None
# example: var = x, term = a, substitution = {}, return {x: a}
    if var in substitution:
        return unify(substitution[var], term, substitution)
    elif ci.is_variable(term) and term in substitution:
        return unify(var, substitution[term], substitution)
    elif occurs_check(var, term, substitution):
        return None
    else:
        new_substitution = {}
        for old_v, old_x in substitution.items():
            new_substitution[old_v] = apply_dict(old_x, {var: term})
        new_substitution[var] = term
        return new_substitution

def single_step(clause1, clause2):
    c1 = rename_clause(clause1, '_1')
    c2 = rename_clause(clause2, '_2')
    for i, term1 in enumerate(c1):
        for j, term2 in enumerate(c2):
            if term1[0] != term2[0] and term1[1] == term2[1]: 
                # P(x) , ~P(a)
                substitution = mgu.MGU(term1, term2)
                # sub = {x: a}
                if substitution is not None:
                    # unify, remove t1 t2

                    for t, lit in enumerate(c2):
                        if t != j : rest.append(lit)
                    # apply sub to rest

                    for lit in rest:
                        new_lit = mgu.apply_dict(lit, substitution)
                        if new_lit not in new_clause:
                            new_clause.append(new_lit)
                    return tuple(new_clause), i , j, substitution
    return None, None, None, None

def solve(kb):

    ## add BFS feature: level, parent_ids
    for clause in kb:
        history.append({'id':len(history)+1,
                        'clause':clause, 
                        'source': 'Original',
                        'level': 0,
                        'parent_ids': None})
    ## i -> new
    curr_level = 0
    i=0
    while i < len(history):
        # use bfs, always try history[i] and history[j],

##### 3.创新点&优化(如果有)
笔者在实现了普通的算法后，在测试中发现，归结会由于穷尽带来很多的重复步骤。如第二个用例输入中，最终结果会达到161行。
在查阅资料和向同学请教学习后，笔者采用bfs类似的思路进行优化。

在solve时，程序会跟之前一样重复穷尽归结，但不会立马打印出来，而是保存在history里，如果检测到归结结果为空了，则开始调用print\_shortest\_path函数向上递归寻找对该步骤其效果的步骤，将其保存起来，另外打印，即可让显示出来的步骤大大减少。

为此，我们在单条日志里引入idx和parent\_id的内容，从而方便从最后一步开始反向寻找。将所有寻找到的放到列表后进行排序，再进行序号的重映射就可以实现上述效果了。

经过优化后，发现三个输出分别是7, 12, 10条，第一第三个和实例输出条数一样，第二个比实例输出少了几条，都是没有用到的前提，也就是说，实际推理可以不用这几个前提。提高了归结效率。
